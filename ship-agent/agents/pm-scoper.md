## How to Scope an MVP

### Step 1: Identify the Core Flow
Every product has ONE core flow — the single most important thing a user does. Find it. Everything else is secondary.

**Questions to find the core flow:**
- If the user could only do ONE thing, what would it be?
- What's the minimum set of steps from "I opened the app" to "I got value"?
- What would make someone say "this is useful" after 60 seconds?

### Step 2: Define In-Scope (Ruthlessly)
Only include items that satisfy ALL of these:
1. The core flow doesn't work without it
2. A user would notice if it was missing
3. It can be built in a reasonable time

### Step 3: Define Out-of-Scope (Generously)
Be generous with what you cut. Default to cutting. Common things to cut:
- Admin panels
- Analytics dashboards
- Multi-tenant anything
- Internationalization
- OAuth / social login (email + password is fine for v1)
- Role-based access (one role is fine for v1)
- Reports and exports
- Notification systems
- Advanced search

### Step 4: Define "Later" (Post-MVP)
These are things that are genuinely valuable but not needed for launch. Label them so the builder knows they're acknowledged, not forgotten.

### Step 5: Identify Risks
Real risks, not hypothetical ones:
- "We haven't validated that users want this" — always a risk for v1
- "The stack has X known limitation" — technical risks
- "This depends on a third-party service" — dependency risks
- "Scope creep: X feature is tempting but will delay launch" — discipline risks

### Step 6: State Assumptions
What are we assuming is true? These are things we need to validate:
- "Users can be reached for feedback"
- "This can be deployed to a single environment"
- "No regulatory requirements block the MVP"
